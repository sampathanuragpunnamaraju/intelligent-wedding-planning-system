# Database Design

# Database Design V1

## Purpose

The database stores:

* User Accounts
* Wedding Events
* Generated Wedding Plans

---

# Entity 1: User

Purpose:

Stores customer account information.

Fields:

id

name

email

password

createdAt

updatedAt

---

Relationship

One User can create multiple Wedding Events.

---

# Entity 2: WeddingEvent

Purpose:

Stores questionnaire responses provided by the customer.

Fields:

id

eventName

eventDate

location

locationDecided

budget

guestCount

venueType

weddingScale

weddingStyle

cateringPreference

guestsTraveling

parkingImportance

dreamWedding

status

createdAt

updatedAt

---

Relationship

Many Wedding Events belong to one User.

---

# Entity 3: GeneratedPlan

Purpose:

Stores reports generated by the Rule Engine.

Fields:

id

budgetReport

guestReport

venueReport

timelineReport

feasibilityScore

generatedAt

---

Relationship

One Generated Plan belongs to one Wedding Event.

---

# Entity Relationships

User (1)
|

|-----> (*) WeddingEvent

---

# Database Strategy

Current MVP focuses on:

* User Management
* Wedding Planning
* Generated Reports

Future entities such as:

* Vendors
* Venues
* Payments
* Notifications
* Bookings

will be added later.

---

# Notes

Generated reports can initially be stored as JSON strings for flexibility and faster development.

Examples:

budgetReport

guestReport

venueReport

timelineReport
